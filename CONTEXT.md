# 增氧机水花巡检系统

使用摄像头巡检养殖场中的增氧机水花状态，识别“设备通电运转但未实际打水”的异常，并将结果上报给后台管理系统。

## 术语

**增氧机**：
用于为养殖水体增氧的桨叶式设备。
_避免_：打氧机、叶轮机

**水花**：
增氧机正常运转时由桨叶击打水面形成的动态喷溅区域。
_避免_：白点、浪花

**预置点**：
摄像头云台预先标定好的观察位置，用于将单台增氧机对应的水花区域稳定带入视野。
_避免_：点位、机位

**识别区域**：
在预置点画面中人为划定、仅用于判断目标水花状态的局部区域。
_避免_：检测框、框选区

**巡检任务**：
按调度策略触发的一轮摄像头逐点检测过程。
_避免_：轮询、扫描

**单点验证**：
以单个预置点、单个目标水花为范围的第一阶段可行性验证。
_避免_：完整试点、全场部署

**标定工具**：
用于建立预置点、目标增氧机和识别区域对应关系的配置工具。
_避免_：部署系统、识别程序

**标定配置包**：
由标定工具导出的、供识别巡检系统导入使用的一份版本化配置产物，且只对应一台摄像头。
_避免_：运行服务、在线接口

**预览流**：
用于人工查看摄像头当前画面的远程视频流。
_避免_：识别输入、设备状态

**视觉状态**：
识别程序输出的结果，取值为“有水花”“无水花”或“不确定”。
_避免_：故障状态、设备状态

**设备运行状态**：
后台管理系统或电控系统给出的增氧机通断电、启停等运行信息。
_避免_：视觉状态

**异常告警**：
视觉状态与设备运行状态交叉比对后得出的待处理异常事件。
_避免_：识别结果、故障结论

**识别巡检系统**：
运行在边缘设备上的、按任务执行预置点巡检与识别上报的后端程序。
_避免_：标定工具、网页后端

**通用识别策略**：
在多个预置点间复用的同一套水花判定逻辑，预置点只提供几何位置与采集配置，不单独维护识别参数。
_避免_：逐点调参、每点一套规则

**全局抖动**：
由摄像头振动、风吹或云台残余晃动导致的整幅画面整体位移，不应被识别为水花动态。
_避免_：水花运动、局部喷溅

**标准化短序列**：
识别巡检系统为单次判定固定采集的一段统一时长、统一采样率、统一帧数的短时序素材。
_避免_：随机多帧、任意抓图

**多特征打分门控**：
综合局部运动、动态占比、持续性、亮度变化、纹理扰动和采集质量的通用判定方式，不依赖单一阈值。
_避免_：单阈值判定、单特征识别

**识别证据**：
用于复核单次视觉状态判定的精简留痕信息，包括代表帧、ROI、关键特征分数和最终结果。
_避免_：仅保留标签、过量调试噪声

**手动触发巡检**：
由开发者或操作员显式发起的一次单摄像头识别执行，用于验证配置消费、采集链路和识别结果。
_避免_：常驻调度、自动轮询

**远程流采样**：
识别巡检系统从摄像头远程视频流中截取标准化短序列作为正式识别输入的采集方式。
_避免_：浏览器冻结帧、临时抓图、并行多输入源

**全局识别配置**：
由识别巡检系统统一维护的采集、等待和判定默认参数，不随单个预置点分别配置。
_避免_：逐点采集参数、逐点门控参数

**复盘素材**：
由识别巡检系统保存的、可回放的标准化短序列及其关键帧，用于离线复核单次识别过程。
_避免_：仅保留结果标签、仅保留控制台文本

**传统识别基线**：
基于 Python、OpenCV 和时序图像规则实现的第二阶段默认识别方案，在其效果不足前不引入训练模型。
_避免_：起步即训练模型、无基线直接上深度学习

**单点识别入口**：
识别巡检系统提供的、针对配置包中单个预置点执行一次完整识别并输出标准结果的最小调用方式。
_避免_：起步即多点巡检、复杂 UI 入口

**标准识别结果**：
单次识别输出的精简结果结构，包含视觉状态、关键分数摘要、证据路径、执行耗时和算法版本。
_避免_：过度细碎字段、海量中间变量

**执行结果**：
描述单次识别流程是否成功完成采集与判定的状态，用于区分流程失败与视觉不确定。
_避免_：与视觉状态混用、所有异常都塞进不确定

**时序投票**：
根据标准化短序列中足够比例的帧级判定结果来输出最终视觉状态的通用判定方式。
_避免_：单帧定结论、瞬时阈值拍板

**帧级二值判定**：
对标准化短序列中的单帧给出“像水花”或“不像水花”的基础判断，供时序投票汇总。
_避免_：单帧三值结论、直接输出最终状态

**加权帧分**：
将局部残余运动强度、动态区域占比和高亮扰动强度按统一权重合成为单帧通过分数的方式。
_避免_：所有特征逐条硬通过、逐点单独权重

## 关系

- **预置点** 与单台 **增氧机** 的 **水花** 形成一对一映射
- **单点验证** 只覆盖一个 **预置点** 和一个目标 **水花**
- **标定工具** 为 **预置点** 和 **识别区域** 建立配置关系
- **标定配置包** 由 **标定工具** 产出并交付给 **识别巡检系统**
- 一个 **标定配置包** 可包含同一台摄像头下的多个 **预置点**
- **预览流** 为 **标定工具** 提供人工观察画面
- **巡检任务** 按顺序访问多个 **预置点**
- 每个 **预置点** 内包含一个 **识别区域**
- **通用识别策略** 在多个 **预置点** 间复用
- **通用识别策略** 需要区分 **全局抖动** 与 **水花** 的局部动态
- **全局抖动** 以整幅画面的整体振动判断，不依赖单独的局部稳定区域配置
- **全局抖动** 先通过整帧对齐抵消，再以 ROI 内残余运动判断 **水花** 动态
- **识别巡检系统** 为每次判定采集一段 **标准化短序列**
- **通用识别策略** 先做 **帧级二值判定**，再做 **时序投票**
- **通用识别策略** 通过 **时序投票** 汇总帧级判定
- **通用识别策略** 采用 **多特征打分门控** 输出 **视觉状态**
- **帧级二值判定** 主要依赖短时序内部的动态对比，而不是长期依赖标定截图
- 第二阶段首版的 **帧级二值判定** 先收敛到局部残余运动强度、动态区域占比和高亮扰动强度 3 个主特征
- 第二阶段首版的 **帧级二值判定** 通过 **加权帧分** 合成单帧结果
- 每次 **视觉状态** 判定都应保留精简的 **识别证据**
- 第二阶段先以 **手动触发巡检** 验证识别闭环
- 第二阶段的正式识别输入采用 **远程流采样**
- 第二阶段先完成白天场景的通用识别闭环，再适配夜间红外
- 单个 **标定配置包** 只提供预置点与几何信息，采集与判定默认参数由 **全局识别配置** 提供
- 第二阶段的 **手动触发巡检** 先只覆盖单个 **预置点** 的一次识别
- 第二阶段会为每次识别保留一份可回放的 **复盘素材**
- 第二阶段默认采用 **传统识别基线**，训练模型属于后备提升路线
- 第二阶段先通过 **单点识别入口** 验证完整识别闭环
- 第二阶段先收敛 **标准识别结果**，再考虑更复杂的结果包装
- **执行结果** 与 **视觉状态** 分开表达，流程失败不等同于视觉不确定
- 第二阶段先将 **执行结果** 收敛为少量固定枚举，而不是复杂错误体系
- 第二阶段按分步实现、分步实测验收推进，每一步过关后再进入下一步
- 第二阶段可用停机状态作为 **无水花** 的可控验证样本来源，不等待真实桨叶断裂事故
- **复盘素材** 的保存不能阻塞在线识别主流程，避免影响单次识别时效
- 在线识别先返回 **标准识别结果**，**复盘素材** 再异步落盘
- 异步生成的 **识别证据** 与 **复盘素材** 必须复用本次识别实际生效的配置快照，不能在 worker 中重新读取已经变化的当前全局配置
- 第二阶段在线识别先只运行白天模式，不提前引入自动昼夜切换
- 第二阶段首版正式远程流输入先锁定 FLV，不提前接入 HLS 自动降级
- 第二阶段首版先信任已验证可用的预置点链路，不额外做独立视觉验位
- 第二阶段首版先复用第一阶段现有标定 JSON 作为配置包原型，不优先重做配置格式
- 第二阶段先在同一仓库内按独立模块实现识别巡检系统，不优先拆成独立仓库
- 第二阶段不优先开发可视化结果页面，当前以命令行输出、日志和复盘素材作为验收方式
- 第二阶段首版按单预置点严格串行执行，不优先做并发识别
- **识别区域** 输出对应目标的 **视觉状态**
- **视觉状态** 与 **设备运行状态** 组合生成 **异常告警**

## 示例对话

> **开发者：** “这个预置点识别失败，是不是说明增氧机坏了？”
> **领域专家：** “不一定。预置点只能先给出该目标的视觉状态，真正的异常告警还要结合设备运行状态判断。”

> **开发者：** “后台显示设备在运行，但画面里没有水花，这算什么？”
> **领域专家：** “这属于视觉状态与设备运行状态不一致，应生成异常告警。”

## 已标记的歧义

- “故障” 在本项目中不能直接等同于 **无水花**。`无水花` 是视觉结论，`故障` 是业务结论。
- “点位” 容易同时指代摄像头预置点、设备编号和后台对象，本文统一使用 **预置点** 表示摄像头观察位。
- 第二阶段的目标是验证 **通用识别策略**，而不是为每个 **预置点** 单独维护一套识别参数。
- 第二阶段的动态判定不能把 **全局抖动** 误判成 **水花**。
- 第二阶段不为单个 **预置点** 增加专门的稳定区域参数，**全局抖动** 直接按整幅画面判断。
- 第二阶段的留痕要以 **识别证据** 为主，既可复核也不能堆成难以使用的调试信息。
- 第二阶段优先验证白天通用识别，夜间红外适配属于同阶段后半段工作。
- 第二阶段可先以约 60% 的 **时序投票** 通过率作为起步标准，后续再逐步收紧到工业化标准。
