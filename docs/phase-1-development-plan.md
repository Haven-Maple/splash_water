# 第一阶段开发顺序与模块清单

## 1. 文档目的

本文档用于明确第一阶段的开发顺序、模块边界、阶段目标和验收方式，指导项目从“方案讨论”进入“可执行开发”。

第一阶段只围绕单点验证展开，不以多预置点、多水花覆盖或完整工业交付为目标。

## 2. 第一阶段目标

第一阶段的目标是完成单个目标水花的识别闭环，形成以下能力：

- 摄像头转到指定预置点
- 抓取该预置点下的当前画面或短时视频流
- 配置并保存单个识别区域
- 对单个目标水花输出视觉状态
- 保存证据截图或片段
- 将结果即时上报或模拟上报

换句话说，第一阶段要证明的是：

`单点标定 + 单点采集 + 单点识别 + 单点上报`

这条链路是可以真实跑通的。

## 3. 第一阶段不做的内容

为了避免目标扩散，第一阶段先不做以下内容：

- 多预置点统一管理
- 多目标轮巡调度
- 单摄像头覆盖数量承诺
- 复杂后台告警编排
- 大规模部署能力
- 白天和夜间共用一套完全统一的识别参数

这些内容可以在第一阶段完成后进入第二阶段逐步补齐。

## 4. 推荐开发顺序

推荐按以下顺序推进：

1. 明确单点配置数据结构
2. 实现单点标定工具
3. 实现摄像头单点采集能力
4. 实现白天单点识别 MVP
5. 将预置点链路接入识别程序
6. 实现结果留存与即时上报
7. 补充夜间红外识别适配

这个顺序的核心逻辑是：

- 先解决“如何拿到正确素材”
- 再解决“如何判断素材里有没有水花”
- 最后再解决“如何让系统稳定运行并输出结果”

## 5. 模块拆分建议

第一阶段建议拆成以下 5 个模块。

### 5.1 配置与标定模块

职责：

- 维护摄像头 ID、预置点 ID、目标名称等基础信息
- 抓取当前预置点画面
- 在画面上框选识别区域
- 保存单点配置

建议输出的配置字段：

- 摄像头 ID
- 预置点 ID
- 目标增氧机 ID 或名称
- 识别区域坐标
- 采集模式
- 采集时长
- 备注信息

建议第一版形态：

- 简单页面或桌面式小工具
- 能查看画面并画框保存即可

第一版不要追求：

- 批量管理
- 权限系统
- 多用户协作
- 复杂配置历史

### 5.2 摄像头控制与采集模块

职责：

- 调用大华接口跳转到指定预置点
- 等待预置点画面稳定
- 抓取单帧图片或 1 至 3 秒短视频流
- 输出识别所需素材

建议优先能力：

- 指定预置点跳转
- 当前帧抓图
- 固定时长短流采集

这一模块是第一阶段最先需要打通的外部依赖。

### 5.3 白天识别模块

职责：

- 基于单个识别区域判断是否存在目标水花
- 输出有水花、无水花或不确定
- 输出置信度和辅助信息

建议第一版先只做白天场景，原因如下：

- 白天可视信息更直接
- 更容易验证 ROI 是否正确
- 更适合快速迭代规则参数

建议第一版优先使用规则型方法，例如：

- 区域内运动强度
- 连续帧亮斑变化
- 高频纹理变化
- 动态区域占比

### 5.4 结果留存与上报模块

职责：

- 保存本次识别的截图、时间戳和结果
- 输出标准化结果对象
- 上报到后台或模拟上报

建议输出字段：

- 摄像头 ID
- 预置点 ID
- 目标 ID
- 识别时间
- 视觉状态
- 置信度
- 素材路径或引用
- 算法版本

第一版即使没有正式后台接口，也应该先做模拟上报，保证链路完整。

### 5.5 夜间红外适配模块

职责：

- 针对夜间红外场景重新评估识别策略
- 调整阈值、判定规则或引入额外特征
- 输出夜间专用的稳定结果

这个模块建议在白天单点闭环完成后再做，不要并行抢跑。

## 6. 每一步的建议产物

### 6.1 第一步：明确单点配置数据结构

产物：

- 一份单点配置 JSON 结构或配置文件格式
- 一份字段说明

目标：

- 统一后续工具、采集、识别和上报之间的数据接口

最低要求：

- 同一个配置能被标定工具保存
- 也能被识别程序读取

### 6.2 第二步：实现单点标定工具

产物：

- 可选择预置点
- 可抓当前画面
- 可框选识别区域
- 可保存配置

目标：

- 让“单点配置”不再依赖手工改坐标

最低要求：

- 能完成一次配置并重复读取

### 6.3 第三步：实现摄像头单点采集能力

产物：

- 一个可调用的采集程序或服务
- 能转预置点
- 能抓图或抓短流

目标：

- 证明能够稳定取得识别素材

最低要求：

- 至少成功完成多次“转动 -> 稳定 -> 采集”

### 6.4 第四步：实现白天单点识别 MVP

产物：

- 一个可输入 ROI 素材并输出视觉状态的识别程序

目标：

- 先在白天场景下区分有水花与无水花

最低要求：

- 对一批白天样本能给出基本可用结果

### 6.5 第五步：接入预置点链路

产物：

- 从“指定配置”到“自动完成采集并识别”的闭环脚本或服务

目标：

- 把标定结果和识别逻辑真正串起来

最低要求：

- 输入一个配置后，能完整跑完一次检测流程

### 6.6 第六步：实现结果留存与即时上报

产物：

- 截图留存
- 日志留存
- 本地结果记录
- 上报接口或模拟上报接口

目标：

- 让结果可以追溯、可展示、可复核

最低要求：

- 每次识别都有结果记录

### 6.7 第七步：补充夜间红外适配

产物：

- 夜间样本测试结果
- 夜间专用参数或规则
- 必要时的独立夜间判定流程

目标：

- 在红外场景下获得基本可用的识别效果

最低要求：

- 夜间样本不至于大面积误判

## 7. 关键依赖关系

第一阶段的依赖关系应明确如下：

- 没有单点配置结构，就不适合并行做标定工具和识别程序
- 没有稳定采集能力，识别程序很容易被伪问题干扰
- 白天识别未闭环前，不建议优先投入夜间优化
- 结果留存和模拟上报应尽早接入，否则难以复盘效果

建议按“配置 -> 采集 -> 识别 -> 上报”的依赖顺序推进。

## 8. 最可能卡住的地方

### 8.1 预置点并不稳定落位

风险：

- 每次转到同一预置点，目标位置略有偏差

影响：

- 识别区域可能框不住目标水花

建议：

- 第一版 ROI 不要框得过小
- 每次采集时保留截图，先观察偏移程度

### 8.2 标定工具做得过重

风险：

- 把大量时间花在页面、配置管理和交互细节上

影响：

- 核心识别闭环迟迟无法验证

建议：

- 第一版工具只做“看画面、框区域、保存配置”

### 8.3 夜间问题拖慢整体进度

风险：

- 红外夜间效果远比白天复杂

影响：

- 第一阶段迟迟无法形成成果

建议：

- 将夜间作为第一阶段后半段专项，不与白天闭环绑死

### 8.4 识别程序被输入不稳定误导

风险：

- 采集到的素材质量不稳定，导致算法问题与采集问题混在一起

影响：

- 调参方向会失真

建议：

- 先固定采集流程，再调识别逻辑

## 9. 第一阶段验收建议

第一阶段建议按“闭环是否成立”验收，而不是按“最终工业化指标”验收。

建议验收项：

- 能否从配置文件读取单点配置
- 能否调用预置点并抓到正确画面
- 能否在固定 ROI 上输出视觉状态
- 能否留存截图和日志
- 能否完成即时上报或模拟上报
- 白天识别是否达到基本可用

夜间能力可作为第一阶段后半段增强项，不建议作为第一版最早验收门槛。

## 10. 当前推荐结论

当前路线是可行的，但推荐用以下顺序来降低风险：

1. 先做单点配置结构
2. 再做最小化标定工具
3. 再做单点采集
4. 再做白天识别 MVP
5. 再接通完整闭环
6. 最后补夜间红外适配

最重要的控制点不是“工具是否完整”，而是“单点闭环是否尽快成立”。
