# 第一、二阶段成果汇报

日期：2026-07-09

## 一、项目阶段结论

当前项目已完成两个可交付的阶段性成果：

1. 第一阶段：完成了面向单台摄像头的网页标定工具。
2. 第二阶段：完成了面向单台摄像头、单个增氧机目标的后端识别程序。

两阶段之间已经形成闭环：

- 第一阶段负责远程连接摄像头、调整云台、保存预置点、框定 ROI、生成标定配置。
- 第二阶段直接消费第一阶段生成的标定配置，执行转预置点、短序列采样、水花识别、结果输出和证据留存。

就当前项目目标来看，这两阶段已经达到“可用、可验证、可继续向后推进”的状态。

## 二、第一阶段成果：网页标定工具

## 1. 阶段目标

第一阶段的目标不是做最终运行平台，而是解决“如何把摄像头目标标定清楚”这个前置问题。

工具当前已经可以支持操作员完成以下工作：

- 连接指定摄像头并检查在线状态
- 获取实时预览画面
- 通过 PTZ 控制摄像头方向和变焦
- 查询、保存、调用预置点
- 在画面稳定后冻结当前帧
- 在冻结帧上框定识别 ROI
- 保存本地标定 JSON 和对应截图

## 2. 当前实际价值

第一阶段的价值主要有三点：

- 把“设备 -> 预置点 -> 目标区域 -> 配置文件”这条链路跑通
- 让后续识别程序不再依赖人工临时判断，而是直接复用标定结果
- 让后续出现识别问题时，可以回看当时的标定截图、ROI 和配置，便于复盘

## 3. 已完成的关键设计收口

当前第一阶段不是原始草案状态，而是经过多轮收口后的可用版本，关键点包括：

- 已移除 `stabilityRoi`，避免维护两套 ROI，降低操作复杂度
- 保留视觉判稳机制，但改为对整幅预览画面做采样
- 保留“命令接受 -> 机械稳定 -> 流追帧 -> 视觉判稳 -> 可抓图”的门控链
- 标定结果已统一落为本地 JSON，可被第二阶段直接读取

这意味着第一阶段已经不只是一个演示页面，而是第二阶段识别链的正式输入工具。

## 4. 当前状态评价

第一阶段当前可以认为：

- 功能已够用
- 联调已跑通
- 能稳定支撑当前项目的标定工作

已知还有一些非阻塞特点，例如预览为了稳定性仍保留了少量延迟，但这不会影响当前标定任务的可执行性。

## 三、第二阶段成果：后端水花识别程序

## 1. 阶段目标

第二阶段的目标，是在不依赖网页前端持续运行的情况下，基于标定结果独立完成识别。

也就是说，第二阶段最终面向的是边缘侧执行逻辑，而不是操作员页面。

当前第二阶段已经实现：

- 读取第一阶段保存的标定 JSON
- 调用摄像头预置点
- 等待画面稳定
- 拉取远程 FLV 短序列
- 执行识别算法
- 输出结构化 JSON 结果
- 异步保存复盘证据

## 2. 当前识别程序的核心特点

当前识别程序不是简单看“亮不亮”或“动不动”，而是围绕真实现场问题，逐步收敛出的门控式识别链。

它的设计重点不是“让某个样本过一次”，而是：

- 复用统一配置
- 避免把反光、波纹、桨叶、画面抖动误识别成水花
- 在白天和夜晚红外两种场景下都能工作
- 在不确定时输出 `undetermined`，而不是强行给错答案

## 四、第二阶段识别算法说明

## 1. 整体执行链路

当前单点识别的实际链路如下：

1. 读取标定配置
2. 转到对应预置点
3. 等待云台稳定和流追帧
4. 拉取固定规格的 FLV 短序列
5. 判断当前画面应走白天识别还是夜晚红外识别
6. 对采样序列做整帧对齐
7. 在 ROI 内提取结构特征和动态特征
8. 先做硬门控，再做加权帧分
9. 对整段短序列做时序投票
10. 输出最终结果，并异步保存复盘证据

这条链路的关键思想是：识别对象不是单帧，而是一段短序列。

## 2. 为什么必须用短序列而不是单帧

如果只看一帧，现场里很多东西都可能被误判：

- 白天的反光水面
- 水波纹
- 夜晚红外下的白色桨叶
- 摄像头轻微晃动

因此程序不是做单帧截图判定，而是采样一小段固定长度的视频序列，再综合判断：

- 是否存在连续的主体结构
- 是否存在局部动态证据
- 这些证据在整段时间内是否足够稳定

这也是当前误判率能显著下降的根本原因。

## 3. 整帧对齐：先处理镜头抖动，再看 ROI

第二阶段里很关键的一步，是在提取 ROI 特征之前，先做整帧对齐。

目的很明确：

- 如果整幅画面一起动，那更可能是镜头晃动
- 如果只有 ROI 内局部区域呈现稳定变化，才更接近真实水花

当前对齐模块会：

- 估计每帧相对首帧的全局位移
- 对位移进行限幅应用
- 保留原始位移、实际应用位移和超限标记

这一步主要是防止把“大风吹动摄像头造成的整体抖动”误当成水花动态。

## 4. ROI 特征提取：不是只看亮，而是看“像不像水花”

当前算法提取的特征可以分成两大类：

### 结构特征

结构特征回答的是：ROI 里是否出现了一个“像水花主体”的区域。

当前重点使用的结构特征包括：

- 最大亮团面积占比
- 亮团是否集中在 ROI 中心
- 亮团是否足够连续，是否碎片化过高
- 亮团是否具有足够的垂直扩展
- 夜晚红外下是否填平了原本桨叶之间的暗缝

### 动态特征

动态特征回答的是：这个结构是不是静止亮物，而不是正在翻动的水花。

当前重点使用的动态特征包括：

- 局部残余运动
- 动态面积占比
- 高亮区域扰动
- 夜晚红外下的面积变化
- 夜晚红外下的形状变化

这一步的核心思想是：水花既要“看起来像水花”，也要“动起来像水花”。

## 5. 白天识别逻辑

白天的主要困难不是看不见目标，而是：

- 水面反光强
- 周边高亮区域多
- 波纹和反光会制造伪动态

因此白天算法最后收敛成了“结构主导，动态辅助”的方向。

当前对白天真实水花的定义，更接近于：

- ROI 中心附近出现大块连续的亮白主体
- 主体具有较强的垂直展开特征
- 不是零散反光碎片
- 至少有一项动态证据成立

换句话说，白天不是看“有没有亮和动”，而是看“有没有像喷溅主体一样的连续亮团”。

## 6. 夜晚红外识别逻辑

夜晚红外的难点和白天完全不同，主要是：

- 画面更模糊
- 噪点更多
- 增氧机桨叶本身就是亮白色
- 单纯靠“白色高亮”很容易把桨叶误判成水花

因此夜晚红外最终采用的是“结构更强主导、动态更弱辅助”的路线。

当前对夜间真实水花的定义，更接近于：

- ROI 中央形成更大的主亮团
- 亮团覆盖更充分
- 亮团在竖向上扩展更明显
- 原本桨叶条状结构之间的暗缝被大量填充
- 再辅以弱动态证据，确认不是静止亮结构

夜晚红外这条线之所以最终能稳定下来，一个非常重要的经验不是单纯调阈值，而是：

- ROI 不能框得太紧
- 必须给增氧机小范围漂移和水花外扩预留余量

## 7. 硬门控 + 加权帧分

当前算法不是所有帧都直接打分，而是先过硬门控。

意思是：

- 如果这一帧连最基本的“水花形状”都不像，就直接判掉
- 只有形态上基本成立的帧，才进入后面的加权评分

这样做的好处是：

- 能显著压住白天反光误报
- 能显著压住夜晚桨叶亮条误报
- 避免后面投票阶段被大量无效帧污染

硬门控通过后，程序再根据各项特征做归一化加权，得到帧级分数，并判断该帧是否通过。

## 8. 时序投票：最终结果由整段序列决定

第二阶段最终结果不是“取最高分那一帧”，而是看整段短序列里有多少帧通过。

最终输出采用三值：

- `has_splash`
- `no_splash`
- `undetermined`

当前逻辑是：

- 通过率足够高，则判定为 `has_splash`
- 通过率足够低，则判定为 `no_splash`
- 中间不确定区间，则判定为 `undetermined`

这个三值策略很重要，因为它允许程序诚实表达“不确定”，而不是硬挤成二分类误报。

## 9. 可靠性门控

除了正常投票，程序还加入了可靠性门控，用来处理一些不该直接给结果的场景，例如：

- 整体位移超限帧占比过高
- 对齐后 ROI 运动并没有明显下降

这类场景下，即使局部特征有一定响应，也会优先压成 `undetermined`。

这样做的目的，是避免镜头整体抖动或者异常流状态把结果带偏。

## 10. 自动白天/夜晚切换

当前第二阶段已经实现白天可见光和夜晚红外的自动切换。

这部分不是简单判断“是不是黑白图”，而是基于全局画面的颜色和通道统计做场景判断。

程序支持三种模式：

- `day_visible`
- `night_ir`
- `auto`

当使用 `auto` 时：

- 如果判断很明确，就直接走对应算法
- 如果判断不够明确，就分别跑白天和夜晚两条链
- 若两条链结果一致，则接受该结果
- 若结果不一致，则返回 `undetermined`

这个设计的意义在于：

- 既实现了一条命令自动适配
- 又避免傍晚切换时段被错误强切

## 五、当前验证结果

## 1. 白天单点验证

当前冻结白天基线结果为：

- 正样本：10 / 10 判定为 `has_splash`
- 负样本：10 / 10 判定为 `no_splash`
- 合计：20 / 20 正确

说明白天识别链已经能够较稳定地区分：

- 真实增氧机水花
- 普通反光、波纹和非目标高亮

## 2. 夜晚红外单点验证

当前冻结夜晚红外基线结果为：

- 正样本：40 / 40 判定为 `has_splash`
- 负样本：10 / 10 判定为 `no_splash`
- 合计：50 / 50 正确

说明夜晚红外这条线已经不再受早期“桨叶误识别为水花”问题的阻塞。

## 3. 自动场景切换

当前自动场景切换能力已经接入主链，支持：

- 手动指定白天
- 手动指定夜晚红外
- 自动识别并切换

从实现和验证结果看，这项能力已经具备实用价值，尤其适合后续真实运行链路。

## 4. 伪多点测试

在单点识别基本稳定后，项目还完成了伪多点测试程序。

该程序会反复执行：

- 先转去过渡预置点
- 再转回真实识别预置点
- 调用既有识别链
- 连续多轮记录结果和耗时

它的作用不是增加新规则，而是验证：

- 多轮调用下识别是否还能稳定触发
- 转动和识别的时序是否可靠
- 后续多点轮询是否具备推进基础

目前从测试结果看，伪多点程序已经达到“可用于下一步参考”的状态。

## 六、当前成果的项目意义

从项目整体推进角度看，第一、二阶段的完成意味着三件事：

1. 标定问题已经有了稳定入口，不再依赖临场记忆和临时手动操作。
2. 单点识别问题已经形成可复用的后端链路，而不是一次性脚本。
3. 后续如果进入真实多点轮询，已经不是从零开始，而是在现有识别主链上继续扩展调度层。

## 七、当前边界与已知非阻塞风险

当前成果是可以交付的，但边界也需要说清楚。

当前未覆盖或仍需后续回归的内容包括：

- 特殊天气下的强反光
- 大风导致的镜头晃动和杂乱水波
- 更复杂的傍晚切换边界场景
- 多摄像头、多点位的正式轮询调度
- 与最终管理后台、数据库、配置同步系统的整合

这些内容属于下一阶段问题，不影响当前第一、二阶段成果成立。

- 第一阶段标定工具已完成并可投入使用
- 第二阶段单点识别程序已完成并达到当前交付目标
- 白天和夜晚红外两条识别线都已有冻结基线
