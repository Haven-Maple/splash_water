# 第二阶段第一版实施方案

## 1. 目标

第二阶段第一版的目标不是完成工业化巡检平台，而是完成一个可独立运行、可复盘、可逐步收紧标准的单点水花识别闭环。

第一版需要证明以下链路成立：

`读取现有标定 JSON -> 指定单个预置点 -> 转到预置点 -> 远程流采样 -> 全局抖动抑制 -> ROI 内帧级判定 -> 时序投票 -> 输出标准识别结果 -> 异步保存复盘素材`

## 2. 范围

### 2.1 本版要做

- 独立于标定工具后端的识别巡检后端
- 与第一阶段同仓分模块实现，不单独拆仓
- 手动触发的单点识别入口
- 复用第一阶段现有标定 JSON 作为配置包原型
- 远程 FLV 流采样
- 白天模式识别
- 基于 Python + OpenCV 的传统图像识别基线
- 全局抖动抑制
- 帧级二值判定
- 时序投票输出三值视觉状态
- 精简标准结果输出
- 异步复盘素材落盘
- 离线回放评测入口

### 2.2 本版不做

- 多预置点连续巡检
- 常驻调度服务
- 自动昼夜切换
- 夜间红外正式识别
- HLS 自动降级
- 可视化结果页面
- 并发识别
- 逐预置点识别参数
- 独立视觉验位
- 训练模型
- 后台管理系统正式对接
- 数据库、配置分发、远程发布

## 3. 系统边界

### 3.1 标定工具系统

职责：

- 配置摄像头预置点
- 框选 ROI
- 保存标定 JSON
- 必要时人工复核

### 3.2 识别巡检系统

职责：

- 读取标定配置
- 控制摄像头转预置点
- 采样标准化短序列
- 执行识别
- 输出结果
- 保存证据和复盘素材

### 3.3 共享对象

两套系统之间只共享标定产物，不共享运行时进程。

当前第一版直接复用第一阶段现有标定 JSON。

## 4. 配置原则

### 4.1 单点配置只负责“看哪里”

单个预置点配置只保留：

- `deviceId`
- `channelId`
- `presetIndex`
- `roi`

必要时可继续保留已有辅助字段，但不把识别参数塞进单点配置。

### 4.2 全局识别配置负责“怎么采、怎么判”

全局识别配置负责：

- 采集时长
- 采样率
- 统一等待策略
- 帧级特征权重
- 帧级通过阈值
- 时序投票通过率
- 复盘素材保存策略

这些参数只能全局调整，不按预置点单独维护。

## 5. 输入输出定义

### 5.1 正式输入

第一版正式输入只采用 FLV 远程流采样。

识别输入是固定规格的标准化短序列，例如：

- 时长：约 2 秒
- 采样率：8 到 12 fps
- 帧数：固定

### 5.2 在线入口

建议入口形式：

```bash
python -m inspector.run_once --config data/calibrations/XXX.json --preset 1
```

含义：

- 指定配置文件
- 指定单个预置点
- 严格串行执行一次识别

### 5.3 离线入口

建议补一个离线回放入口：

```bash
python -m inspector.replay_once --input path/to/sequence
```

目的：

- 使用已保存复盘素材重复跑同一算法
- 用于调试、复查和回归验证

## 6. 识别策略

### 6.1 总体原则

- 采用通用识别策略
- 不按预置点单独调识别参数
- 以动态特征为主，亮度特征为辅
- 不使用标定截图作为长期识别基准
- 标定截图只用于留痕和人工复核

### 6.2 全局抖动抑制

第一步先判断整幅画面是否存在全局抖动：

- 根据整帧估计主位移
- 对相邻帧做整帧对齐
- 计算对齐后的 ROI 残余运动

目标：

- 把风吹镜头、残余晃动、整体抖动从水花动态里分离出去

### 6.3 帧级二值判定

单帧只输出二值结果：

- 像水花
- 不像水花

第一版先收敛到 3 个主特征：

- 局部残余运动强度
- 动态区域占比
- 高亮扰动强度

这 3 个特征通过全局统一权重合成为单帧分数，不要求逐条硬通过。

### 6.4 时序投票

最终结果不由单帧决定，而由整段短序列投票决定。

建议首版规则：

- 根据帧级通过比例做时序投票
- 起步可先以约 60% 作为序列通过标准
- 后续再逐步收紧到更高标准

### 6.5 最终视觉状态

只有执行成功时才输出视觉状态：

- `有水花`
- `无水花`
- `不确定`

说明：

- `不确定` 代表素材拿到了，但视觉上还不能稳定下结论
- 流失败、预置点失败、帧不足不算 `不确定`，而算执行失败

## 7. 执行结果与视觉状态分离

建议执行结果先收敛成少量固定枚举：

- `success`
- `preset_failed`
- `stream_failed`
- `insufficient_frames`
- `detect_error`

只有 `success` 时才继续看 `visualState`。

## 8. 标准识别结果

第一版先输出精简标准结果，不做复杂包装。

建议字段：

- `executionResult`
- `visualState`
- `scoreSummary`
- `evidencePaths`
- `timing`
- `algorithmVersion`

其中：

- `scoreSummary`：局部运动、动态占比、高亮扰动、投票通过率等摘要
- `evidencePaths`：代表帧、短序列、调试图等路径
- `timing`：转预置点、采样、识别耗时

## 9. 识别证据与复盘素材

### 9.1 识别证据

每次识别保留精简证据：

- 代表帧
- ROI 信息
- 关键分数摘要
- 最终结果

要求：

- 能复核
- 不堆过量调试噪声

### 9.2 复盘素材

保存可回放短序列或帧序列文件，用于离线复跑。

要求：

- 在线识别优先返回标准结果
- 复盘素材异步落盘
- 复盘素材保存不得阻塞在线识别主流程

## 10. 样本策略

第一版样本不等待真实桨叶断裂事故。

建议样本来源：

- `有水花`：真实运行中的增氧机
- `无水花`：停机状态
- `不确定`：强反光、强风抖动、采样异常、夜间噪点重等边界情况

说明：

第二阶段先验证“明显有”和“明显无”是否能稳定区分。

## 11. 实施顺序

建议严格按以下 4 步推进，每步完成后立刻实测，过关再进入下一步。

### 第一步：配置消费 + 单点识别入口

目标：

- 能读取第一阶段标定 JSON
- 能指定单个预置点运行
- 能输出基础执行结果

验收：

- 命令能正确读取配置
- 能找到目标预置点
- 能输出结构化结果

### 第二步：远程流采样 + 复盘素材落盘

目标：

- 能从 FLV 流拿到标准化短序列
- 能在不阻塞主链路的前提下异步保存复盘素材

验收：

- 能稳定拿到固定时长、固定帧数素材
- 在线结果返回不被落盘拖慢

### 第三步：整帧对齐 + 帧级二值判定

目标：

- 完成全局抖动抑制
- 完成 3 主特征加权帧分
- 输出帧级通过结果

验收：

- 大风或整体晃动不会明显被当成水花
- 明显有水花时帧级结果多数通过
- 停机无水花时帧级结果多数不通过

### 第四步：时序投票 + 标准识别结果

目标：

- 由帧级结果汇总出最终视觉状态
- 输出标准识别结果
- 补齐证据和耗时信息
- 固化本次识别配置快照，保证异步证据生成与主链路判定使用同一套参数

验收：

- 有水花样本大多数判为 `有水花`
- 停机样本大多数判为 `无水花`
- 边界样本允许输出 `不确定`

第四步建议按以下子项落地。

#### 11.4.1 投票输入

时序投票直接消费第三步已经产出的结构化结果，不重新发明一套并行判断：

- `framePass`
- `weightedScore`
- `framePassCount`
- `sampledFrameCount`
- `overflowFlags`
- `globalMotionExceeded`
- `preAlignmentRoiMotionMean`
- `postAlignmentRoiMotionMean`

原则：

- 只有 `executionResult = success` 才进入时序投票
- `preset_failed`、`stream_failed`、`insufficient_frames`、`detect_error` 仍然属于流程失败，不映射成 `不确定`
- 第四步不新增逐预置点参数，仍然只允许使用全局识别配置

#### 11.4.2 首版投票规则

建议首版先采用简单、可解释、便于复核的三区间规则：

- `passRatio = framePassCount / sampledFrameCount`
- 当 `passRatio >= sequenceVoteThreshold` 时输出 `有水花`
- 当 `passRatio <= 1 - sequenceVoteThreshold` 时输出 `无水花`
- 介于两者之间时输出 `不确定`

在当前起步阈值 `sequenceVoteThreshold = 0.6` 下，首版等价于：

- `passRatio >= 0.6`：`有水花`
- `passRatio <= 0.4`：`无水花`
- `0.4 < passRatio < 0.6`：`不确定`

这样做的目的：

- 先把“明显有”和“明显无”稳定拉开
- 给边界样本留出缓冲区，避免为了首版通过率过早把灰区样本硬判成黑白两类
- 保持规则足够简单，便于后续现场回看日志和调参

#### 11.4.3 可靠性门控

第四步不能只看投票比例，还要补一层轻量可靠性门控，避免明显不可信的序列被硬出结论。

首版建议门控：

- 若 `sampledFrameCount` 明显低于目标帧数，继续维持现有 `insufficient_frames`
- 若全局位移超限帧占比过高，则优先输出 `不确定`
- 若对齐后 ROI 残余运动没有明显下降，且 `globalMotionExceeded = true`，则优先输出 `不确定`

说明：

- 这里的门控只用于把“不可信但拿到素材”的情况压到 `不确定`
- 不要把门控扩展成复杂规则树，第四步仍然以“时序投票为主、可靠性门控为辅”
- 这些门控参数仍然归入全局识别配置，不按预置点拆分

#### 11.4.4 标准结果补齐

第四步完成后，标准结果至少应稳定包含：

- `executionResult`
- `visualState`
- `scoreSummary.framePassRatio`
- `scoreSummary.framePassCount`
- `scoreSummary.sampledFrameCount`
- `scoreSummary.sequenceVoteThreshold`
- `scoreSummary.globalMotionExceeded`
- `scoreSummary.overflowFrameCount`
- `timing.detectMs`
- `evidencePaths.representativeFramePath`
- `evidencePaths.replaySequencePath`
- `evidencePaths.replayMetadataPath`
- `evidencePaths.debugImagePath`

要求：

- `visualState` 只在 `executionResult = success` 时出现有效值
- 输出文案与结构化字段语义一致，不出现“执行失败却给视觉结论”的混用
- 控制台日志保留足够复查信息，但不输出海量逐帧噪声

#### 11.4.5 异步证据配置快照

第四步必须把当前已经暴露的小风险一起收口：

- 异步 replay worker 不能再依赖“启动时重新读取当前 `recognition_v1` 配置”
- 必须使用“本次主链路实际生效的识别配置快照”来生成代表帧、调试图和离线复跑上下文

建议实现：

- 在主链路进入检测阶段后，立即冻结一份 `effectiveRecognitionConfig`
- 将这份快照写入 handoff 或单独写成 `recognition-config.snapshot.json`
- replay worker 只读取这份快照，不再读取可变的 `local_config.json`
- `metadata.json` 中记录 `algorithmVersion`、配置快照摘要和关键阈值，保证复盘时能解释“当次为什么这么判”

这样可以避免：

- 主链路已经按旧参数完成打分，但 worker 因为读取了新配置而生成不一致证据
- 线上返回结果与稍后落盘的调试图、代表帧、离线复跑结论不一致

#### 11.4.6 第四步专项验收

除原有“有水花 / 无水花 / 不确定”验收外，第四步建议补 3 条专项验收：

- 修改 `local_config.json` 后立即触发一次识别，再在主返回后、worker 落盘前改一次配置，确认落盘证据仍与主结果一致
- 强风或明显镜头晃动样本中，允许更多输出 `不确定`，但不应大量误判为 `有水花`
- 控制台或日志中能直接看出本次 `passRatio`、投票阈值、是否触发 `globalMotionExceeded`

## 12. 第一版验收标准

第一版先不强行写死成“准确率必须达到多少百分比”，而采用可操作通过条件：

- 能读取现有标定 JSON 并指定单个预置点运行
- 能成功转到预置点并拿到 FLV 标准化短序列
- 能输出 `执行结果` 和 `视觉状态`
- 对明显有水花样本，大多数判为 `有水花`
- 对停机无水花样本，大多数判为 `无水花`
- 对边界样本允许输出 `不确定`
- 能保留复盘素材，且不阻塞主识别流程

## 13. 下一步建议

第一版通过后，再进入第二阶段后半段：

- 白天多预置点复用验证
- 夜间红外识别适配
- HLS 兼容补充
- 多点巡检入口
- 自动昼夜切换评估
- 更严格量化验收指标
